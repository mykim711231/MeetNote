# 회의록 앱 최적 기술 스택 조사 (2026)

> 조사일: 2026-06-17 · 대상: MeetNote(개인용 무료 로컬 PWA) 및 향후 품질 업그레이드 경로
> 결론 요약: **지금은 "브라우저 내장(무료·로컬)"이 정답. 품질이 필요해지면 STT만 단계적으로 교체**한다.

---

## 0. 한눈에 — 3단계 경로

| 단계 | STT | 요약/할일 | 비용 | 프라이버시 | MeetNote 적용 |
|---|---|---|---|---|---|
| **A. 무료·로컬 (현재)** | Web Speech API (ko-KR) | 규칙 기반(로컬) | 0 | 최상(기기 내) | ✅ 구현됨 |
| **B. 온디바이스 고품질** | Whisper/Moonshine (WebGPU/WASM) | 규칙 기반 또는 로컬 LLM | 0 | 최상(기기 내) | 모델 수십~수백 MB 다운로드 |
| **C. 클라우드 최상** | Deepgram·AssemblyAI·Whisper API / (한국어) 리턴제로·클로바 | Claude/GPT map-reduce | 종량 | 외부 전송 | 서버·키 필요(무료 원칙 이탈) |

개인용 무료 앱이라면 **A 유지가 최적**, 정확도 불만 시 **B**(여전히 무료·오프라인), 팀/업무용으로 커지면 **C**.

## 1. STT(음성→텍스트) 엔진 비교

### 1-1. 클라우드 API (영어권 벤치마크, 2026)
- **AssemblyAI Universal-2**: WER ≈ **8.4%**, Whisper 대비 환각 30%↓, 영숫자·포맷 정확도 우수. 화자분리·감정·PII·주제탐지 등 "오디오 인텔리전스" 풍부. 배치 $0.12/h, 스트리밍 $0.15/h
- **OpenAI GPT-4o Transcribe**: WER ≈ **8.9%**, 고정확도. $0.006/min (4o-mini-transcribe $0.003/min)
- **Deepgram (Nova/Flux)**: 정확도 2위권이지만 **실시간 스트리밍·속도 최강**. 배치 $0.0043/min, 스트리밍 $0.0077/min → 보이스 에이전트/실시간에 최적
- **NVIDIA Canary/Parakeet**: Open ASR Leaderboard WER ≈ **5.63%**(셀프호스트)
- **ElevenLabs Scribe v2**: 다국어 실시간

> 선택 가이드: **실시간=Deepgram**, **셀프호스트=Whisper/NeMo**, **트랜스크립트 인텔리전스(요약·할일까지)=AssemblyAI**, **다국어 실시간=ElevenLabs**.

### 1-2. 한국어 STT
- **리턴제로(ReturnZero)**: 경쟁사 대비 오류율 **35~46%↓**, 2.5배 빠르고 최대 3배 저렴 — 한국어 정확도/가성비 선두권
- **네이버 클로바(CLOVA Speech)**: 한국어 인식률 최상위, 1시간+ 장문 회의 처리 안정적(클로바노트 실적)
- **구글 STT**: 특정 케이스 한국어 정확도 낮다는 실사용 평가 다수
- 벤치마크 출처: [Awesome-Korean-Speech-Recognition](https://github.com/rtzr/Awesome-Korean-Speech-Recognition), [리턴제로 한국어 벤치마크](https://blog.rtzr.ai/korean-speechai-benchmark/)

> 한국어 회의록 품질이 핵심이면 **리턴제로 또는 클로바**가 Whisper/구글보다 유리.

### 1-3. 온디바이스/브라우저 (무료·오프라인·프라이버시)
- **Moonshine Web**: WebGPU + ONNX Runtime Web + Transformers.js로 **브라우저 내 실시간**. Whisper tiny 대비 연산 5×↓·동등 WER, **75ms 지연**, RoPE로 가변 길이(Whisper의 30초 윈도 제약 없음). 데이터 외부 전송 0
- **Whisper WebGPU / whisper-web (xenova)**: Transformers.js로 브라우저에서 Whisper 실행, 클라우드급 품질 + 완전 로컬
- **whisper.cpp**: Python 무의존 C/C++ 포트, Apple Metal·CUDA·Vulkan 가속. 데스크톱/네이티브 래핑 시 유리
- **Vosk**: 경량 오프라인(저사양/임베디드)

> MeetNote의 "무료·로컬·오프라인" 원칙을 **깨지 않고** 정확도를 올리는 유일한 길 = **Moonshine/Whisper WebGPU(Transformers.js)**. 단, 모델 다운로드(수십~수백 MB)와 WebGPU 지원 브라우저 필요.

## 2. 실시간 스트리밍 아키텍처 (클라우드 STT 채택 시)

- **지속 WebSocket** 세션: 시작 시 인증 + 모델 파라미터(sample rate, encoding, channels), `interim_results`/`diarization` 플래그
- **오디오 청킹**: 마이크 입력을 **100ms 프레임**으로 전송(저지연). 네트워크 절감 위해 250ms로 늘리면 지연↑
- **부분 결과(interim) → 확정(final)** 흐름으로 자막 즉시 표시
- 서버가 프레임 처리 → 트랜스크립트를 실시간 회신
- 참고 오픈소스: [WhisperLiveKit](https://github.com/QuentinFuxa/WhisperLiveKit)(Simul-Whisper·AlignAtt 정책, 스트리밍+화자분리), [Baseten fastest Whisper](https://www.baseten.co/blog/the-fastest-whisper-transcription-with-streaming-and-diarization/)

> 현재 MeetNote의 Web Speech API는 이 구조를 브라우저가 내부적으로 처리(서버 불필요). 클라우드로 가면 위 WebSocket 파이프라인을 직접 구축해야 함.

## 3. 화자 분리(Speaker Diarization)

- **본질적으로 어려움**: 화자 수 미상, 중간 합류, 태그-트랜스크립트 정렬 — 실시간일수록 난도 급증
- **클라우드**: AssemblyAI·Deepgram이 단일 스트림 화자분리 제공(가장 실용적)
- **온디바이스**: pyannote 등 WASM 포팅 가능하나 무겁고 정확도 한계
- **개인용 권장**: 자동 분리 대신 **수동 화자 라벨**(현 MeetNote 방식)이 비용 0·충분. 자동화는 클라우드 단계에서.

## 4. 요약·할 일 추출

### 4-1. 규칙 기반 (현재, LLM 불필요)
- 빈도 기반 추출 요약 + 액션 패턴 매칭. 비용 0·오프라인. 품질은 "키워드 수준".

### 4-2. LLM (클라우드 단계)
- **모델**: Claude Sonnet 4.6(기본 일꾼, 1M 컨텍스트) / Opus 4.7(이사회·전략 등 정밀) / Haiku 4.5(다건 배치·저비용). 대안 Gemini 2.5 Pro
- **액션아이템 추출 정확도 ≈ 90~95%** (깨끗한 트랜스크립트 + 구조화 프롬프트 기준; 잡음·교차발화·전문용어 시 저하)
- **긴 회의 = map-reduce**: 15분 청크별 요약 → 최종 통합 프롬프트
- **프롬프트 5분류**: ① 결정사항(+결정자·조건) ② 액션아이템(+담당·기한) ③ 미해결 질문 ④ 공유된 핵심 정보 ⑤ 요약
- AssemblyAI LLM Gateway처럼 트랜스크립트에 20+ 프런티어 모델(Claude/GPT/Gemini) 직접 적용 가능

## 6. MeetNote 권장 결론

1. **현행 유지가 정답**: 개인용 무료·로컬 목표엔 Web Speech API + 규칙 요약이 비용 0·프라이버시 최상으로 최적.
2. **정확도 업그레이드(무료 유지)**: 다음 큰 한 수는 **Transformers.js + Whisper/Moonshine(WebGPU)** 온디바이스 전사 옵션 추가. 원칙(무료·오프라인) 불변, 트레이드오프는 모델 다운로드.
3. **품질 최우선으로 전환 시**: 한국어는 **리턴제로/클로바**, 다국어·인텔리전스는 **AssemblyAI**, 실시간은 **Deepgram**. 요약은 **Claude(map-reduce, 5분류 프롬프트)**. 단 서버·키·비용 발생 → "무료" 포기 결정 필요.
